자바 웹 어플리케이션

WAS에 설치되어 동작하는 어플리케이션
자바 웹 어플리케이션에는 HTML, CSS, 이미지, 자바로 작성된 클래스(Servlet도 포함됨, package, 인터페이스 등), 각종 설정 파일 등이 포함됩니다.

내가 인터넷에서 사용하는 쇼핑몰 블로그가 웹 어플리케이션이라고 말 할 수 있다.
앞에서 만든건 서블릿으로만 이루어진 아주아주아주 간단한 웹 어플리케이션이라고 할 수 있다.


자바 웹 어플리케이션 폴더 구조

  • WEB-INF
    이 폴더 안에는 web.xml이 있다.
    web.xml은 배포 기술자라고도 하고, 웹 어플리케이션에 대한 정보를 다 가지고 있는 파일이라고 할 수 있다.
    WEB-INF폴더 하위 폴더로 lib라는 폴더,classes 라는 폴더가 존재함

  • web.xml
    https://lordofkangs.tistory.com/35
    WEB-INF 폴더 밖은 접근 가능하지만, WEB-INF 폴더 안의 서블릿에는 접근이 불가능하다
    이때 WEB-INF 폴더 안의 서블릿에 접근 하기 위해서 진짜 경로를 숨기고 가짜 경로를 만들어 외부 접근으로부터
    서버로직을 보호할 수 있다.

  • lib폴더
    각종 jars 파일

  • classes 폴더
    java 패키지, 컴파일 된 class 들..
    내가 작성한 서블릿 파일이 들어감

  • 리소스들
    각종 폴더, 이미지, 다양한 리소스들

Servlet

자바 웹 어플리케이션의 구성 요소 중 동적인 처리를 하는 프로그램의 역할
동적으로 만들어 낸다는 것은!
이미 응답할 페이지를 만들어서 가지고 있는게 아니라 , 요청이 들어왔을때 프로그램이 실행되면서
응답할 코드를 만들어 내고 그 코드로 응답을 하는 것을 말한다.

서블릿을 정의해보면 서블릿(servlet)은 WAS에 동작하는 JAVA 클래스입니다.

서블릿은 HttpServlet 클래스를 상속받아야 합니다.

서블릿과 JSP로부터 최상의 결과를 얻으려면, 웹 페이지를 개발할 때 이 두 가지(JSP, 서블릿)를 조화롭게 사용해야 합니다.

예를 들어, 웹 페이지를 구성하는 화면(HTML)은 JSP로 표현하고, 복잡한 프로그래밍은 서블릿으로 구현합니다.


###Servlet 작성방법
두가지가 있는데

  • Servlet 3.0 이상일때 어노테이션을 사용한다
  • Servlet 3.0 미만에서는 web.xml 파일에 직접 적어서 한다

현재 실무에서는 웹을 개발할때 서블릿을 써서 개발하지 않음
조금 더 편하게 사용할 수 있게 도와주는 다양한 프레임 워크를 사용한다.

하지만 프레임 워크들도 서블릿 없이는 동작 할 수 없기 때문에 서블릿의 기본적인 작성법이나, 서블릿의 라이프사이클을 이해할 줄 알면 좋다.


Servlet 3.0 이상 방법에서 작성해보기

새로운 다이나믹 웹 프로젝트를 만들때 Dynamic web module versiond을 3.1로 설정 하였다.

  • HttpServletRequest request, HttpServletResponse response
    클라이언트가 요청 -> 서버가 응답
    클라이언트가 요청할때 서버가 요청을 받아내는 객체와 응답을 하기 위한 객체를 두개를 자동을 만든다.
    요청에 대한 객체는 HttpServletRequest, 응답에 관한 객체는 HttpServletResponse에 담아둔다고 생각하면 된다.

  • response.setContentType(“text/html;charset=utf-8”);
    파일의 형식이 뭔지 알려준다
    위는 html임을 확인 할 수 있다.

  • PrintWriter out = response.getWriter();
    response 객체에 getWriter() 메서드가 존재하는데,
    getWriter()라는 메서드를 수행하면 PrintWriter 객체가 반환 된다

  • @WebServlet(“/Ten”)
    url 주소값이다
    위의 코드의 주소가 http://localhost:8080/exam31/Ten로 설정 됨을 확인 할 수 있다


Servlet 3.0 미만의 방법에서 작성해보기

새로운 다이나믹 웹 프로젝트를 만들때 Dynamic web module versiond을 2.5로 설정 하였다.
3.1과는 다르게 web.xml이 자동으로 만들어진다 (3.1은 프로젝트 만들면서 따로 체크해줘야함)

  • 3.0미만의 버전은 어노테이션이 없다

아래는 web.xml의 일부이다

url이 /ten이라는 요청이 들어오면
-> URL mapping에서 찾아낸다 url이 존재한다면 (존재안하면 404)
-> 을 확인하고 이름이 TenServlet과 같은 이름을 에서 확인한다
-> 서블릿을 찾아서 실행시킬 서블릿이 누군지 에서 찾을 수 있다

3.0이상의 버전에서는 어노테이션이 위의 같은 역할을 한다

서블릿이 언제 생성되고 이런 메서드들이 언제 호출이 되는지 알아볼 수 있다.

LifecycleServlet 생성
init 호출
service 호출

위의 결과가 콘솔에 출력됨
최초로 LifecycleServlet를 호출시 객체가 만들어지면서
생성자가 만들어진다 따라서 생성자에 넣어준 메세지가 출력된다 -> LifecycleServlet 생성


service 호출
브라우저 새로고침시 저 한줄만 나온다
서블릿은 서버에 객체를 여러번 만들지 않음
요청된 객체가 메모리에 있는지 없는지 확인하고, 있으면 service라는 메서드만 호출한다


destroy 호출
서블릿이 수정되면 현재 메모리에 올라가 있는 서블릿 객체가 더 이상 사용될 수 없다
이때 destroy 되면서 메모리에 객체를 삭제함

  • 클라이언트의 요청이 Post일 경우에는 자신이 가지고 있는 doPost(request, response)를 호출

HttpServletRequest와 HttpServletResponse (객체)

웹 브라우저는 도메인과 포트번호를 이용해서 서버에 접속
-> path 정보, 클라이언트의 IP, 클라이언트의 다양한 정보를 포함한 요청정보를 서버에 전송
-> 클라이언트로부터 요청이 들어오면
-> WAS는 HttpServletRequest와 HttpServletResponse 객체를 생성한다
-> HttpServletRequest는 요청할떄 가지고 들어온 다양한 정보를 담는다
-> HttpServletResponse는 요청을 보낸 클라이언트에게 보낼 정보를 담는다
-> 이렇게 생성된 두개의 객체를 요청정보의 path로 매핑된 서블릿에 전달
-> 이렇게 전달한 객체는 service(), doGet(), doPost() 같은 메서드의 파라미터로 전달되어서 사용한다

  • HttpServletRequest
    http프로토콜의 request정보를 서블릿에게 전달하기 위한 목적으로 사용합니다.
    헤더정보, 파라미터, 쿠키, URI, URL 등의 정보를 읽어 들이는 메소드를 가지고 있습니다.
    Body의 Stream을 읽어 들이는 메소드를 가지고 있습니다.
  • HttpServletResponse
    WAS는 어떤 클라이언트가 요청을 보냈는지 알고 있고, 해당 클라이언트에게 응답을 보내기 위한 HttpServleResponse객체를 생성하여 서블릿에게 전달합니다.
    서블릿은 해당 객체를 이용하여 content type, 응답코드, 응답 메시지등을 전송합니다.
